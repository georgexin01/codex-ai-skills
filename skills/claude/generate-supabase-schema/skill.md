# Generate Supabase Schema (generate-supabase-schema)

Generate SQL schema for the Supabase database based on existing TypeScript type definitions.

## Description

Reads TypeScript type files from `apps/web-antd/src/types/` (already generated by `/generate-store`) and generates complete SQL to set up:
- Public schema tables (shared auth/RBAC infrastructure)
- Business schema tables (project-specific data)
- Foreign key relationships, indexes, triggers
- Row Level Security (RLS) policies
- Seed/dummy data (optional)

**IMPORTANT:** This skill generates SQL and shows it to the user for review. The user will copy-paste it into Supabase Studio SQL Editor (`studio.aisolo.vip` → SQL Editor) to execute.

## Pipeline Position

```
/analyze-schema → /generate-store → /generate-supabase-schema → /generate-views
```

## Multi-Project Architecture

Supabase uses a **two-layer** schema architecture:

### Layer 1: `public` schema (shared auth/RBAC)

Shared across ALL projects. Contains authentication infrastructure.

| Table | Purpose | Column Style |
|---|---|---|
| `public.project` | Project registry — each row = a project with its own schema | **snake_case** |
| `public.role` | Roles per project (admin, agent, etc.) — has `project_id` FK | **snake_case** |
| `public."user"` | Auth users — linked to `auth.users` via `auth_id`, scoped by `project_id` + `role_id` | **snake_case** |

**Functions:**
- `custom_access_token_hook(event jsonb)` — JWT hook that injects `user_role`, `project_id`, `role_level` into JWT claims
- `get_current_project_id()` — reads `project_id` from JWT
- `get_current_role()` — reads `user_role` from JWT
- `get_current_role_level()` — reads `role_level` from JWT

**IMPORTANT:** Public schema tables use **snake_case** columns (e.g., `auth_id`, `project_id`, `is_delete`, `created_at`). This matches the VPS Supabase and Supabase conventions.

### Layer 2: Business schema (project-specific data)

Each project has its own schema (e.g., `quizLaa`, `wms`, `test_school`).

| Example | Purpose | Table Name | Column Style |
|---|---|---|---|
| `"quizLaa".users` | App-level user records | **snake_case** | **camelCase** |
| `"quizLaa".lessons` | Business-specific tables | **snake_case** | **camelCase** |
| `"quizLaa".question_answers` | Multi-word table names | **snake_case** | **camelCase** |
| `"quizLaa".attachment` | File attachment records | **snake_case** | **camelCase** |

**IMPORTANT:**
- **Table names** use **snake_case** (e.g., `question_answers`, `attachment`)
- **Column names** use **camelCase** (e.g., `isDelete`, `createdAt`, `lessonId`)
- This matches the TypeScript naming convention for columns while keeping table names consistent with PostgreSQL conventions.

### Supabase Storage Setup

When a schema needs file uploads, create a storage bucket named after the schema:

```sql
-- Bucket name = schema name
INSERT INTO storage.buckets (id, name, public) VALUES ('quizLaa', 'quizLaa', true);

-- RLS policies for the bucket
CREATE POLICY "Authenticated upload" ON storage.objects FOR INSERT TO authenticated WITH CHECK (bucket_id = 'quizLaa');
CREATE POLICY "Public read" ON storage.objects FOR SELECT TO public USING (bucket_id = 'quizLaa');
CREATE POLICY "Authenticated update" ON storage.objects FOR UPDATE TO authenticated USING (bucket_id = 'quizLaa');
CREATE POLICY "Authenticated delete" ON storage.objects FOR DELETE TO authenticated USING (bucket_id = 'quizLaa');
```

Upload path convention: `{YYMM}/{YYMMDD-HHMMSS}-{filename}` (e.g., `2603/260323-120231-photo.jpg`).

The `attachment` table tracks all uploaded files with soft-delete support. See `utils/upload.ts` for the upload utilities.

### Dual User Table Pattern

There are TWO user tables serving different purposes:

| Table | Purpose | Used By |
|---|---|---|
| `public."user"` | Auth/RBAC — linked to `auth.users`, has `role_id` FK, used by JWT hook | Login flow, JWT claims, RLS |
| `"schema"."users"` | Business data — app-level user management, has simple `role` string | UI CRUD, business logic, stores |

The business `users` table (in the schema) is what the Pinia store manages. The `public."user"` table is managed by the auth flow.

### PostgREST Schema Exposure

The business schema must be exposed via PostgREST. For Supabase CLI projects, update `config.toml`:

```toml
# D:\supabase-project\supabase\config.toml
[api]
schemas = ["public", "graphql_public", "{{SCHEMA}}"]
```

Then restart: `cd D:\supabase-project && supabase stop && supabase start`

**Note:** The `supabase_vector` container sometimes gets stuck. If start fails, run `docker rm -f supabase_vector_supabase-project` then retry.

Also enable the JWT hook in config.toml if not already:

```toml
[auth.hook.custom_access_token]
enabled = true
uri = "pg-functions://postgres/public/custom_access_token_hook"
```

## TypeScript → PostgreSQL Type Mapping

| TypeScript Type | PostgreSQL Type | Notes |
|---|---|---|
| `string` (id field) | `uuid PRIMARY KEY DEFAULT gen_random_uuid()` | Primary key |
| `string` | `text` | General strings |
| `string` (with max length) | `varchar(n)` | Length-limited strings |
| `number` (integer) | `integer` | Whole numbers |
| `number` (decimal) | `numeric(10,2)` | Money/decimal values |
| `boolean` | `boolean NOT NULL DEFAULT false` | |
| enum (TypeScript enum) | `text NOT NULL CHECK ("col" IN (...))` | Use CHECK constraint |
| `string[]` / `any[]` | `jsonb NOT NULL DEFAULT '[]'::jsonb` | Arrays stored as JSONB |
| `any` (object) | `jsonb NOT NULL DEFAULT '{}'::jsonb` | Objects stored as JSONB |
| `createdAt: string` | `timestamptz NOT NULL DEFAULT now()` | |
| `updatedAt: string` | `timestamptz NOT NULL DEFAULT now()` | |
| FK field (`xxxId: string`) | `uuid REFERENCES "schema"."table"("id")` | Foreign key |
| Optional one-to-one FK | `uuid REFERENCES ...` plus partial unique index on active rows | `CREATE UNIQUE INDEX ... WHERE "xxxId" IS NOT NULL AND "isDelete" = false` |

## SQL Generation — Two Parts

### Part 1: Public Schema (run once per Supabase instance)

Only generate this if the public tables don't exist yet. Skip if restoring from VPS backup.

```sql
-- ============================================================
-- PUBLIC SCHEMA — Shared Auth/RBAC Infrastructure
-- ============================================================

-- Project table
CREATE TABLE IF NOT EXISTS public.project (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  schema_name text NOT NULL,
  description text,
  status text NOT NULL DEFAULT 'active',
  is_delete boolean NOT NULL DEFAULT false,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

COMMENT ON TABLE public.project IS 'Project table - each project corresponds to an independent schema';
COMMENT ON COLUMN public.project.schema_name IS 'Schema name for the project, e.g., quizLaa';

-- Role table
CREATE TABLE IF NOT EXISTS public.role (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  description text,
  level integer NOT NULL DEFAULT 100,
  is_system boolean NOT NULL DEFAULT false,
  is_delete boolean NOT NULL DEFAULT false,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  project_id uuid REFERENCES public.project(id)
);

COMMENT ON TABLE public.role IS 'Role table - lower level means higher permission';
COMMENT ON COLUMN public.role.level IS 'Permission level: 0=highest, 100=lowest';

-- User table (auth layer)
CREATE TABLE IF NOT EXISTS public."user" (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  auth_id uuid,
  project_id uuid REFERENCES public.project(id),
  role_id uuid NOT NULL REFERENCES public.role(id),
  name text NOT NULL,
  email text,
  phone text,
  avatar_url text,
  status text NOT NULL DEFAULT 'active',
  is_delete boolean NOT NULL DEFAULT false,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT user_email_or_phone CHECK (email IS NOT NULL OR phone IS NOT NULL)
);

COMMENT ON TABLE public."user" IS 'User table - linked to auth.users';
COMMENT ON COLUMN public."user".auth_id IS 'Reference to Supabase Auth user';

-- JWT Hook
CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event jsonb)
RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER SET search_path TO 'public' AS $$
DECLARE
  active_project_id UUID;
  user_project_id UUID;
  user_role TEXT;
  user_role_level INT;
  claims JSONB;
BEGIN
  claims := event->'claims';
  active_project_id := (event->'claims'->'user_metadata'->>'active_project_id')::UUID;

  IF active_project_id IS NULL THEN
    SELECT u.project_id INTO active_project_id
    FROM public."user" u
    WHERE u.auth_id = (event->>'user_id')::UUID
      AND u.is_delete = false AND u.status = 'active'
    ORDER BY u.created_at ASC LIMIT 1;
  END IF;

  SELECT u.project_id, r.name, r.level
  INTO user_project_id, user_role, user_role_level
  FROM public."user" u
  JOIN public.role r ON u.role_id = r.id
  WHERE u.auth_id = (event->>'user_id')::UUID
    AND u.project_id = active_project_id
    AND u.is_delete = false AND u.status = 'active';

  IF user_project_id IS NOT NULL THEN
    claims := jsonb_set(claims, '{project_id}', to_jsonb(user_project_id));
  END IF;
  IF user_role IS NOT NULL THEN
    claims := jsonb_set(claims, '{user_role}', to_jsonb(user_role));
    claims := jsonb_set(claims, '{role_level}', to_jsonb(user_role_level));
  END IF;

  RETURN jsonb_set(event, '{claims}', claims);
END;
$$;

-- Helper functions
CREATE OR REPLACE FUNCTION public.get_current_project_id() RETURNS uuid
  LANGUAGE sql STABLE SECURITY DEFINER AS $$
  SELECT (auth.jwt() ->> 'project_id')::uuid;
$$;

CREATE OR REPLACE FUNCTION public.get_current_role() RETURNS text
  LANGUAGE sql STABLE AS $$
  SELECT auth.jwt() ->> 'user_role';
$$;

CREATE OR REPLACE FUNCTION public.get_current_role_level() RETURNS integer
  LANGUAGE sql STABLE SECURITY DEFINER AS $$
  SELECT COALESCE((auth.jwt() ->> 'role_level')::int, 100);
$$;

-- RLS for public tables
ALTER TABLE public.project ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.role ENABLE ROW LEVEL SECURITY;
ALTER TABLE public."user" ENABLE ROW LEVEL SECURITY;

CREATE POLICY "authenticated_access" ON public.project FOR ALL TO authenticated USING (true) WITH CHECK (true);
CREATE POLICY "authenticated_access" ON public.role FOR ALL TO authenticated USING (true) WITH CHECK (true);
CREATE POLICY "authenticated_access" ON public."user" FOR ALL TO authenticated USING (true) WITH CHECK (true);
```

### Part 2: Business Schema (per project)

This is the main output of the skill — generated from TypeScript types.

```sql
-- ============================================================
-- BUSINESS SCHEMA: {{SCHEMA}}
-- ============================================================
CREATE SCHEMA IF NOT EXISTS "{{SCHEMA}}";

-- Shared trigger function
CREATE OR REPLACE FUNCTION "{{SCHEMA}}"."set_updated_at"()
RETURNS TRIGGER AS $$
BEGIN
  NEW."updatedAt" = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Tables (ordered by dependency — parents first)
CREATE TABLE IF NOT EXISTS "{{SCHEMA}}"."{{tableName}}" (
  "id" uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  -- entity-specific columns (camelCase)
  "isDelete" boolean NOT NULL DEFAULT false,
  "createdAt" timestamptz NOT NULL DEFAULT now(),
  "updatedAt" timestamptz NOT NULL DEFAULT now()
);

-- Trigger, indexes, RLS per table...

-- RLS SELECT policies should check read permission only. Do not include
-- `"isDelete" = false` in SELECT policies; normal frontend/store queries
-- must explicitly pass `.eq('isDelete', false)` when fetching active rows.
DROP POLICY IF EXISTS "{{tableName}}_select" ON "{{SCHEMA}}"."{{tableName}}";
CREATE POLICY "{{tableName}}_select" ON "{{SCHEMA}}"."{{tableName}}"
  FOR SELECT TO authenticated
  USING ("{{SCHEMA}}".authorize('{{RESOURCE}}', 'read'));

-- Permissions
GRANT USAGE ON SCHEMA "{{SCHEMA}}" TO anon, authenticated;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA "{{SCHEMA}}" TO authenticated;
GRANT SELECT ON ALL TABLES IN SCHEMA "{{SCHEMA}}" TO anon;
ALTER DEFAULT PRIVILEGES IN SCHEMA "{{SCHEMA}}"
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO authenticated;
ALTER DEFAULT PRIVILEGES IN SCHEMA "{{SCHEMA}}"
  GRANT SELECT ON TABLES TO anon;

-- Seed: register project
INSERT INTO public.project (name, schema_name, description)
VALUES ('{{PROJECT_NAME}}', '{{SCHEMA}}', '{{PROJECT_DESCRIPTION}}');

-- Seed: roles for this project
INSERT INTO public.role (name, description, level, is_system, project_id)
VALUES
  ('admin', 'Administrator', 20, true, (SELECT id FROM public.project WHERE schema_name = '{{SCHEMA}}')),
  ('agent', 'Agent', 60, false, (SELECT id FROM public.project WHERE schema_name = '{{SCHEMA}}'));
```

## Seed Data: Auth Users (for login)

When generating seed data, also generate a separate migration for auth users. Auth users require **3 inserts** across different tables:

### Step 1: Insert into `auth.users`

```sql
DO $$
BEGIN
  IF NOT EXISTS (SELECT 1 FROM auth.users WHERE email = '{{EMAIL}}') THEN
    INSERT INTO auth.users (
      instance_id, id, aud, role, email,
      encrypted_password, email_confirmed_at,
      created_at, updated_at, is_sso_user,
      confirmation_token, recovery_token,
      raw_app_meta_data, raw_user_meta_data
    ) VALUES (
      '00000000-0000-0000-0000-000000000000',
      gen_random_uuid(), 'authenticated', 'authenticated',
      '{{EMAIL}}',
      crypt('123456', gen_salt('bf')),
      now(), now(), now(), false, '', '',
      '{"provider":"email","providers":["email"]}',
      '{}'
    );
    -- IMPORTANT: Fix NULL string columns (GoTrue requires empty strings, not NULL)
    UPDATE auth.users SET
      email_change = '', email_change_token_new = '',
      email_change_token_current = '', phone_change = '',
      phone_change_token = '', reauthentication_token = ''
    WHERE email = '{{EMAIL}}';
  END IF;
END $$;
```

**CRITICAL GOTCHA:** After INSERT, must UPDATE NULL string columns (`email_change`, `phone_change`, etc.) to empty strings `''`. GoTrue throws `"converting NULL to string is unsupported"` if these are NULL.

### Step 2: Insert into `auth.identities`

```sql
INSERT INTO auth.identities (
  id, provider_id, user_id, identity_data, provider,
  last_sign_in_at, created_at, updated_at
)
SELECT
  u.id, u.id, u.id,
  jsonb_build_object('sub', u.id::text, 'email', u.email),
  'email', now(), now(), now()
FROM auth.users u
WHERE u.email = '{{EMAIL}}'
  AND NOT EXISTS (
    SELECT 1 FROM auth.identities i WHERE i.user_id = u.id AND i.provider = 'email'
  );
```

### Step 3: Link to `public."user"`

```sql
INSERT INTO public."user" (auth_id, project_id, role_id, name, email) VALUES (
  (SELECT id FROM auth.users WHERE email = '{{EMAIL}}'),
  (SELECT id FROM public.project WHERE schema_name = '{{SCHEMA}}'),
  (SELECT id FROM public.role WHERE name = '{{ROLE}}' AND project_id = (SELECT id FROM public.project WHERE schema_name = '{{SCHEMA}}')),
  '{{NAME}}',
  '{{EMAIL}}'
);
```

### Auth user ON CONFLICT note

Do NOT use `ON CONFLICT (email) DO NOTHING` — `auth.users` has a partial unique index `WHERE is_sso_user = false`, so the ON CONFLICT clause fails. Use `IF NOT EXISTS` pattern instead.

## Workflow

### Phase 1: Check if public tables exist

Ask: "Is this a fresh Supabase instance, or do you already have public tables (project, role, user) set up?"
- Fresh → generate Part 1 + Part 2
- Already set up → generate Part 2 only + seed project/roles

### Phase 2: Analyze Types & Generate SQL

1. Read type files from `apps/web-antd/src/types/`
2. Parse interfaces for columns, types, constraints
3. Identify FK relationships, determine creation order
4. Generate SQL (Part 1 if needed + Part 2)
5. Present for review

### Phase 3: User runs SQL in Studio

Tell user: "Copy the SQL and paste into Supabase Studio SQL Editor (studio.aisolo.vip → SQL Editor → New query → Paste → Run)"

### Phase 4: PostgREST config (new schema only)

If this is a new schema, remind user to update `config.toml` schemas and restart Supabase CLI.

### Phase 5: Auth users migration

Generate a separate migration file for auth users (auth.users + auth.identities + public."user"). This enables login.

### Phase 6: Save all SQL as migration files

Save all generated SQL to `apps/web-antd/src/sql/migrations/` with numbered prefixes:
- `001_public_schema.sql` — skip if public tables exist
- `002_{{schema}}_schema.sql` — business tables + register project
- `003_{{schema}}_seed.sql` — dummy data
- `004_{{schema}}_permissions.sql` — permissions + authorize functions
- `005_{{schema}}_auth_users.sql` — auth.users + identities + public."user" link

## Notes

1. **Always show SQL before user runs it** — never auto-execute
2. **Public tables = snake_case**, business tables = **camelCase** (quoted identifiers)
3. **CHECK constraints over PG enums** — easier to modify
4. **Order tables by dependency** — parents before children
5. **FK display fields** (e.g., `lessonTitle`) are NOT columns — they come from PostgREST joins
6. **Do NOT include FK display fields** in CREATE TABLE
7. The business `users` table is separate from `public."user"` — different purposes
8. `public."user"` must be quoted because "user" is a PostgreSQL reserved word

## Applying SQL on LOCAL Docker Supabase

When the target is a local Docker stack (not Studio / remote MCP), do NOT inline SQL
with `psql -c` — PowerShell mangles double-quoted camelCase identifiers. Use a file:

```bash
docker ps --format "{{.Names}}" | grep supabase          # discover container name
docker cp migration.sql supabase_db_local-supabase:/tmp/m.sql
docker exec supabase_db_local-supabase psql -U postgres -d postgres -f /tmp/m.sql
```

**RLS + permissions are a DUAL grant.** An `authorize('resource','action')` RLS policy
only passes if the role ALSO has a matching row in the `permissions` table. Generating
the `FOR INSERT` policy is not enough — seed the `create`/`read`/`update`/`delete`
permission rows per role too. Missing `create` rows are the usual 403 cause.

**Once a table has live data**, never DROP+CREATE — add a new numbered corrective
migration. Verify the next number against `ls migrations/ | sort | tail -1` to avoid
collisions. See [VBEN_SUPABASE_LOCAL_LESSONS](../VBEN_SUPABASE_LOCAL_LESSONS.md) for the
full local-Docker gotcha checklist.
